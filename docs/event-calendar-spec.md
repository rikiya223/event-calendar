# イベントカレンダー Webシステム 仕様書

> 世界中のイベント（スポーツ・音楽・展覧会・映画・ゲームなど）を一覧・発見できるカレンダー形式のWebサービス。
> 本書はMVP（初回リリース）を中心に、将来拡張（Phase2/3）も含めた設計方針をまとめたもの。

最終更新: 2026年6月14日 / ステータス: 本番公開済み（付録B＝MVP実装反映、付録C＝公開・データ拡充・UX改善の反映）

---

## 1. プロダクト概要

### 1.1 目的
世界中で日々開催される多様なイベント情報を、ユーザーが「カレンダー形式」で手軽に発見・収集できる一般公開サービスを提供する。

### 1.2 対象ユーザー
一般公開サービス。ログインなしでも閲覧でき、ログインするとブックマークや投稿などの機能が使える。

### 1.3 対象地域
まずは日本国内を対象とする。将来的に多言語・海外対応へ拡大する設計とする。

### 1.4 対象イベントカテゴリ（初期）
スポーツ / 音楽・ライブ / 展覧会・美術館 / 映画・エンタメ / ゲーム・発売日 / 祝日・カレンダー行事 / その他。
カテゴリは今後も増え続ける前提で、親子構造（大分類・中分類）で無限に拡張できる設計とする。

---

## 2. 設計上の重要方針

本プロジェクトの難所は以下の3点であり、それぞれに対する基本方針を最初に定める。

### 2.1 「無限に増えるカテゴリ」への対応
カテゴリ固有の色を割り当てる方式は、数が増えると破綻する（人が識別できる色は7〜8色まで）。
そのため色は「大分類のみ」が持ち、中分類・小分類は親の色を継承し、アイコンとラベルで区別する。

### 2.2 「同じイベントの名寄せ（重複統合）」への対応
同一イベントが複数ソースから取り込まれる前提で、`events`（統合済みの正規レコード）と `event_sources`（各ソースの生データ）を1対多で分離する。これにより統合・分離・再処理が後から柔軟にできる。

### 2.3 「情報をどう集めるか（収集の網羅性）」への対応
収集手段は4系統（公式API・スクレイピング・AIエージェント・ユーザー投稿）を組み合わせる。
ただし自動収集は運用リスクが高いため、MVPでは手動登録とユーザー投稿に絞り、自動化はPhase2以降に段階導入する。

## 3. 機能要件

### 3.1 MVPに含める機能
1. 月・週・日のカレンダー表示（切り替え可能）
2. イベント詳細画面（日時・会場・チケット情報・公式リンク）
3. カテゴリ・地域によるフィルター
4. ブックマーク機能（要ログイン）
5. 管理画面での手動イベント登録
6. ユーザーによるイベント投稿＋管理者の承認フロー
7. 機械的な重複チェック（同日・同会場・類似タイトルを警告）
8. Googleカレンダーへの追加

### 3.2 将来拡張（参考）
- Phase2: AIによる名寄せ判定、自動スクレイピング、AIエージェント収集、プッシュ通知
- Phase3: パーソナライズ推薦、多言語・海外対応、ネイティブアプリ、外部API連携拡大、タイムライン／リスト表示の拡充

## 4. UI / UX 仕様

### 4.1 基本方針
- メインターゲットはスマートフォン（モバイルファースト）。PCにも対応。
- デザインの雰囲気は「やわらか・親しみやすい」パステル系。

### 4.2 主要画面
| 画面 | 内容 |
|------|------|
| カレンダー一覧 | 月／週／日表示。上部にカテゴリフィルターのピル。下部にタブナビ（カレンダー／リスト／保存済み／マイページ）。 |
| イベント詳細 | カテゴリカラーのヒーロー背景。日時・会場・チケット情報・地図・アクションボタン（カレンダー追加／ブックマーク／チケット購入）。 |
| 管理画面 | 手動登録フォーム、ユーザー投稿の審査キュー、重複チェック警告。 |
| マイページ | ブックマーク一覧、投稿履歴、設定。 |

### 4.3 カラー設計
- 大分類カテゴリにのみ色を割り当てる（例：スポーツ＝緑、音楽＝ピンク、展覧会＝アンバー、映画＝赤、ゲーム＝緑系、全体軸＝紫）。
- 中分類・小分類は親カテゴリの色を継承し、アイコンと名称で区別する。

### 4.4 カレンダー上のイベント表示
- 「すべて表示」時は、イベント件数が膨大になるため日付ドットは出さない（または件数バッジのみ）。
- フィルター適用時のみ、絞り込んだカテゴリの色ドットを表示する。
- 月カレンダーは「概観」、実際の発見はリスト／検索が主役という役割分担とする。

## 5. データモデル

### 5.1 全体構成
統合済みイベント（`events`）を中心に、ソース・会場・カテゴリ・開催回・出演者・ユーザー関連テーブルが紐づく。

### 5.2 主要テーブル

#### events（イベント：統合済みの正規レコード）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid (PK) | |
| canonical_title | string | 正規版タイトル |
| description | text | 説明文（AI要約・リライト後を想定） |
| confidence_score | int | 信頼度スコア（自動公開／審査の判定に使用） |
| status | string | published / pending_review / rejected |
| venue_id | uuid (FK) | 開催地 |
| starts_at | timestamp | 開始日時 |
| ends_at | timestamp | 終了日時 |

#### event_sources（ソース：各取得元の生データ）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid (PK) | |
| event_id | uuid (FK) | 統合先イベント |
| source_type | string | official_api / scraping / ai_agent / user / manual |
| source_url | string | 取得元URL |
| raw_payload | json | 生データ（再処理用に保持） |
| trust_weight | int | このソースの信頼度の重み |

#### categories（カテゴリ：親子構造）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid (PK) | |
| parent_id | uuid (FK) | 親カテゴリ（自己参照） |
| name | string | カテゴリ名 |
| color_key | string | 色キー（大分類のみ保持） |
| icon | string | アイコン |

#### venues（会場）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid (PK) | |
| name | string | 会場名 |
| address | string | 住所 |
| lat / lng | float | 緯度・経度 |

#### event_occurrences（開催回：1イベント＝複数日程に対応）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid (PK) | |
| event_id | uuid (FK) | |
| starts_at / ends_at | timestamp | 各回の日時 |

#### performers（出演者・チーム）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid (PK) | |
| name | string | アーティスト名・球団名など |
| type | string | 種別 |

#### users / bookmarks / event_submissions（ユーザー関連）
- `users`: id, email, display_name
- `bookmarks`: id, user_id (FK), event_id (FK)
- `event_submissions`: id, user_id (FK), status, payload

### 5.3 多対多の中間テーブル（ER図には未記載だが実装で必要）
- `event_categories`（イベント ⇔ カテゴリ：1イベントが複数カテゴリを持てる）
- `event_performers`（イベント ⇔ 出演者：1イベントに複数出演者）

### 5.4 信頼度スコアの考え方
各ソースに `trust_weight`（公式API＝高、スクレイピング＝中、ユーザー投稿・AI収集＝低）を持たせ、イベント全体の `confidence_score` を算出する。

| スコア | 扱い |
|--------|------|
| 高（公式API由来など） | 自動承認・即公開 |
| 中（スクレイピング由来など） | 重複チェックのみ通す |
| 低（ユーザー投稿・AI収集） | pending_review として人の審査へ |

これにより、人の審査を「全件」ではなく「怪しいものだけ」に絞れる。

## 6. 情報収集アーキテクチャ

### 6.1 全体フロー
4系統の入力ソースが「統合インジェスト層」に集約され、その後は単一パイプラインに合流する。

```
① 公式API ─┐
② スクレイピング ─┤
③ AIエージェント ─┼→ 統合インジェスト層 → AI整形・正規化 → 人の審査 → イベントDB → 公開
④ ユーザー投稿 ─┘
```

### 6.2 各ソースの方針
- **① 公式API（最優先・最も信頼できる）**: TMDb（映画）など無料公式APIを積極活用。日本のチケット系（ぴあ・e+・ローチケ）は公開APIが乏しく、提携／アフィリエイト経由が中心になる現実を踏まえる。
- **② スクレイピング（最もグレー・要注意）**: robots.txt遵守、利用規約確認、事実情報（日時・場所）と説明文（著作権あり→AIで要約・リライト）の線引きを徹底する。
- **③ AIエージェント（「浅く広く」と好相性）**: 個別スクレイパーを書く代わりにAIに巡回・構造化させる。初期の網羅性向上に有効だがコスト・精度管理が必要。
- **④ ユーザー投稿（抜け漏れの補完）**: 小規模・地域イベントを補完。承認制が前提。

### 6.3 合流後の処理
AI整形・正規化パイプラインで、日時／会場の正規化、カテゴリ自動付与、重複検出・名寄せ、欠損補完を行う。
信頼度スコアにより自動公開／審査を振り分ける。

> 注: MVPでは自動収集（②③）とAI整形は実装しない。手動登録（source_type=manual）とユーザー投稿（user）のみを `event_sources` に格納し、将来の自動収集と自然に共存できる構造だけ用意しておく。

## 7. 名寄せ（重複統合）ロジック

全イベント同士を比較すると件数の2乗で爆発するため、段階的に候補を絞る4ステップ構成とする。

### STEP 1: ブロッキング（粗い絞り込み）
開催日±1日以内かつ同エリアの候補だけを抽出し、比較対象を数件に減らす。DBの日付・地理インデックスで高速化する。

### STEP 2: 機械的スコアリング（AI不使用）
タイトル類似度（表記揺れを吸収）・開始時刻の差・会場一致度を点数化。明確に同一／明確に別物はここで確定する。大半のケースはここで決着。

### STEP 3: AI判定（グレーゾーンのみ）— ※Phase2
STEP 2で中間スコアになったものだけLLMに渡し、文脈・常識で同一性を確信度つきで判定する（例：「大相撲六月場所」と「大相撲夏場所」）。

### STEP 4: 確信度で3分岐
- 高 → 自動統合（既存eventに source を追加）
- 中 → pending_review（人の審査キューへ）
- 低 → 新規イベントとして作成

### 7.1 統合時の正規版（canonical）選択
複数ソースで値が食い違う場合の採用方針：
- **MVP**: 信頼度が最も高いソースの値を丸ごと採用（シンプル・予測可能）。
- **将来**: フィールドごとに最良ソースを選ぶ（タイトルは公式、説明文はユーザー投稿、画像は別ソースなど）。

> MVPでは STEP 1〜2（機械的処理）のみ実装。STEP 3（AI判定）はPhase2。MVPの重複チェックは「同日・同会場・類似タイトルを管理者に警告する」レベルで十分。

## 10. 主要課題と対策（まとめ）

| 課題 | 対策方針 |
|------|----------|
| 情報収集の網羅性 | 公式API・RSS・外部APIを段階統合。ユーザー投稿で補完。 |
| 重複・品質管理 | 機械的重複チェック（MVP）→ AI名寄せ（Phase2）の2段階。信頼度スコアで審査を最小化。 |
| 著作権・利用規約 | robots.txt遵守、公式API優先、説明文はAI要約・出典明示。 |
| コミュニティ品質 | ユーザー投稿は承認制。スパム対策・通報機能。 |
| カテゴリの無限増加 | 親子構造で対応。色は大分類のみ。 |

## 付録: MVP実装の最初の一歩

1. データモデルの骨格（5章）をPostgreSQLのDDLに落とす。`events` / `event_sources` 分離と `categories` 親子構造は最初から入れる。
2. 管理画面の手動登録フォームを作り、数十件のイベントを実際に登録してデータ感を掴む。
3. カレンダー表示とフィルターをつなぎ、最小の閲覧体験を完成させる。
4. ブックマーク・ユーザー投稿・承認フローを追加。
5. Googleカレンダー連携を最後に追加。

> 自動収集（スクレイピング・AIエージェント・AI名寄せ）は、MVPで実データと使われ方を観察した後にPhase2として設計・実装する。

# 付録B: 実装反映（2026年6月13日時点）

実装の過程で本仕様から**変更・追加**した点の記録。仕様本文は当初のまま残し、差分を本章に集約する。

## B-1. 確定した技術スタック（8章の具体化）

| レイヤー | 採用 |
|----------|------|
| フロント/バック | Next.js 16（App Router）+ React 19 + TypeScript + Tailwind CSS v4 |
| ORM | Prisma 6（v7はdatasource設定の破壊的変更があるため6に固定） |
| DB・認証 | **Supabase に一本化**（PostgreSQL + Auth。将来の画像保存も Supabase Storage を想定） |
| 認証方式 | メール＋パスワード（パスワード再設定フロー実装済み。Googleログインは未実装） |
| AI | Anthropic API（AI抽出。`AI_INGEST_MODEL` で切替、既定 claude-opus-4-8） |

## B-2. データモデルの変更（5章からの差分）

- **日時は `event_occurrences` に一本化**。`events.starts_at / ends_at` は持たない（二重持ちを解消）。単発イベントも開催回1件を作る。カレンダー表示・重複チェックは常に開催回を参照。
- `venues.region`（都道府県）を追加 — 地域フィルター用。
- `event_submissions.payload` に取得元メタを追加：`origin`（user / rss / ai）、`sourceName`、`sourceUrl`。
- `event_sources.source_url` は**冪等キー**を兼ねる（例：`caogov:holiday:2026-07-20`、`ical:<UID>`、TMDb作品URL）。再取り込みしても重複しない。
- カテゴリ初期データ：大分類7種＋中分類（**「表彰・賞レース」を追加**。賞レース・表彰式イベント用）。

## B-3. MVPに前倒しで実装した機能（3.2のPhase2相当）

仕様では Phase2 とした自動収集を、**安全な範囲で前倒し実装**した。いずれも秘密鍵（`INGEST_SECRET`）付きAPIルートで、cron から定期実行できる。

| 経路 | ルート | 信頼度の扱い |
|------|--------|--------------|
| 祝日（内閣府CSV・キー不要） | `/api/ingest/holidays` | 公式＝**自動公開**（trust 90） |
| TMDb 映画公開日 | `/api/ingest/tmdb` | 公式＝自動公開（trust 85）※B-6の商用制限に注意 |
| 汎用 iCal（.ics） | `/api/ingest/ical?url=...` | trust 70。`publish=0` で審査キュー行きも可 |
| RSS/Atom フィード | `/api/ingest/rss?url=...` | **審査キュー行き**（半自動）。日本語日付を機械抽出、日付不明の記事は除外 |
| AI抽出（Claude） | `/api/ingest/ai?url=...` | **審査キュー行き**（半自動）。ページ本文から構造化抽出。要 `ANTHROPIC_API_KEY` |

- RSS/AI は既存の**承認フロー（重複警告つき）に合流**する＝6章の「人の審査」をそのまま活用。
- 承認時に取得元に応じて `source_type` / 信頼度を自動設定（RSS=OFFICIAL_API 60、AI=AI_AGENT 30、ユーザー投稿=USER 30）。出典URLもイベントに引き継ぐ。
- 推奨実行頻度：祝日=週1、TMDb=日1、iCal/RSS=日1〜数回。

## B-4. 画面・UXの追加（4章からの差分）

- **ホーム（/）**: ヒーロー＋検索＋カテゴリタイル＋近日開催のランディング。
- **さがす（/explore）**: 検索・人気タグ・カテゴリタイル・地域フィルター・近日開催・おすすめレール。
- **カレンダー（/calendar）**: 月表示は「カレンダー＋選択日のイベント一覧」の2ペイン構成（ダッシュボード型）。月セルはPCでイベント名表示／スマホは色ドット。テキスト検索は今日以降を全期間横断。注目イベントのヒーロー、近日開催リストを併設。
- **イベント詳細**: カテゴリ色ヒーロー、開催回ごとのGoogleカレンダー追加、地図埋め込み、出演者、**共有ボタン**、**出典・リンク欄**、ブックマーク。
- **マイページ**: ブックマーク一覧＋**投稿履歴（状態バッジつき）**。
- **管理画面**: 手動登録（**複数開催回・出演者・都道府県対応**）、編集・削除、審査キュー（**取得元バッジ・元記事リンク・重複警告**）、登録済み一覧に取得元バッジ。
- **認証UX**: ログイン後の元画面復帰（`?next=`）、パスワード表示切替、パスワード再設定。
- **ナビ**: PC=左サイドバー、スマホ=上部バー＋下部タブ＋投稿FAB。
- **その他UX**: ローディングスケルトン、保存トースト、カスタム404/エラー画面、ページ別タイトル、フォーカスリング（a11y）、日本時間明記。
- **デザインシステム**: `docs/design-system.md` のトークン（primary #004ac6 等）＋ Material Symbols に統一。

## B-5. 管理者アクセス制御（追加仕様）

- 管理者は環境変数 `ADMIN_EMAILS`（カンマ区切り）で指定。一致するログインユーザーのみ `/admin` と管理系サーバーアクションを実行できる（ページ・アクション両方でチェック）。

## B-6. データ利用ポリシー（10章の具体化）

- **商用利用OK（無料）**: 政府・自治体オープンデータ（CC BY系。出典表示必須）、内閣府祝日、Wikidata（CC0）、各団体の公開iCal/RSS（事実情報）。
- **無料だが非商用限定**: **TMDb**（商用は別途契約が必要）、connpass無料枠。→ 収益化する場合はTMDb取り込みを停止または契約。
- まとめ系サイト（アートアジェンダ等）の収集は行わない（編集された商用DBのため）。一次情報（各館・各団体の公式発表）のみ扱う。
- 日時・会場・タイトル等の**事実情報**を扱い、説明文は要約・リライトする（著作権方針）。フッターに出典クレジット（TMDb・内閣府）を表示。

## B-7. ジャンル拡充の方針（1.4の補足）

- **スポーツ試合**: Jリーグ等の**公式配布iCal**を `/api/ingest/ical` で取り込み（最優先）。公式icsが無い競技は手動/AI抽出。
- **美術館の特設展示**: 各館公式のRSS・お知らせをRSS/AI経路で取り込み→審査。
- **賞レース・表彰式**: 件数が少ないため手動登録中心（カテゴリ「表彰・賞レース」）。将来はWikidata連携も検討。

## C-9. カレンダー画面のUX
- **ドリルダウン絞り込み**: 大分類を選ぶと、その中分類が出てさらに絞り込める。
- **複数選択**: カテゴリ・地域とも複数選択可（URLはカンマ区切り）。地域は「追加」プルダウン＋✕付きチップ、カテゴリはピルのトグル。
- **絞り込み結果**: カテゴリ/地域で絞ると、該当イベントを**日付順の横カルーセル（矢印送り）**で「直近のイベント」欄に表示（カレンダーの上）。注目イベントは絞り込み中も表示し続けて視線を固定。
- レイアウト: 検索バー → 注目イベント → カテゴリ → 直近イベント → カレンダー。地域は目立たないプルダウンに。「期間」クイック絞り込みは廃止。
- カード: カテゴリ色の左アクセントバー／淡い色面・柔らかい影・ホバーで浮く、に統一。
- **絞り込みの“リロード感”の解消**: ルートの `loading.tsx`（全画面スケルトン）が毎回フラッシュしていたため削除し、絞り込みリンクに `scroll={false}` を付与（前の表示を保持・先頭ジャンプなし）。なお1操作あたり約200〜400msのサーバー往復は残る（完全クライアント側フィルタは今後の検討）。
- トップ（`/`）は `/explore` と役割重複のため廃止し `/calendar` へリダイレクト。PCではフィルタ行を折り返し表示。
